# Performance Prediction for NLP Tasks: Methods, Features, and Scenarios


Predicting how well a model will perform on a new task for a given language is a challenging yet crucial problem in multilingual NLP. Often we want to estimate performance in zero-shot settings – for example, when a model fine-tuned on one language (or a set of languages) is applied to a different target language without additional training. This guide explores the diverse methods available for performance prediction, including the algorithms, features, and scenario-specific techniques. By understanding these methods, even a novice can choose an appropriate strategy to estimate model accuracy or F1 on a new language-task combination. We cover scenarios ranging from having closely related language data to having no direct information, and we discuss how to leverage large language models (LLMs) in zero-shot cases. Throughout, we highlight practical features and tools in Python that can help implement these approaches.

## Common Scenarios for Zero-Shot Performance Prediction
When faced with predicting performance for a target language and target model on some task, you might find yourself in one of several scenarios with respect to available information. Key scenarios include:
- Similar Language & Similar Model Data Available: Performance is known for a closely related language (or languages) using a similar model architecture.
- Distant Language & Model Data Available: Only performance data from languages or models that are quite different from the target are available.
- Same Language & Different Model Size: Performance is known for the same language and task but on a similar model of a different size (e.g. a smaller or larger variant).
- Same Language & Different Model Family: Performance is known for the same language and task but on a different model architecture or family (e.g. a previous generation model).
- No Direct Data (Cold Start): No performance data exists for the target language or model on the task – only other languages or models (or tasks) have been observed.

Additionally, there are task variation scenarios where we might have performance information on other tasks in the same language or with the same model. For example, perhaps we know how the model performs on Part-of-Speech tagging in the target language but need to predict its Named Entity Recognition accuracy. Each scenario may call for a combination of methods to make the best prediction.

## Feature-Based Regression Approaches
One fundamental approach is to treat performance prediction as a regression problem: we gather features that describe the language, model, and task, and train a predictive model (regressor) to output the expected performance metric (accuracy, F1, etc.). This approach has been successfully used in recent research. Key aspects include:
- Designing Informative Features: Choose features that correlate with model performance. Common features include:
    - Pre-training Data Size: The amount of the model’s pre-training data that was in the target language (or a proxy like the size of that language’s Wikipedia). This is often a strong predictor – languages with larger training corpora generally yield better zero-shot results.
    - Linguistic/Typological Features: Encodings of the language’s properties (e.g. word order, morphology, orthography). Typological similarity between the fine-tuning language (pivot) and the target language can be predictive. For instance, if the target language is linguistically similar to the source (pivot) language used to train the model on the task, the model likely transfers better. These features can be obtained from databases like WALS or URIEL (e.g. via the lang2vec library) which provide vectors for properties like syntax, phonology, and genealogy.
    - Lexical/Vocabulary Overlap: The overlap in vocabulary or subword units between the target and the language(s) the model was fine-tuned on. A high subword overlap indicates the model’s tokenizer is well-suited to the target language, which has a positive influence on zero-shot transfer. For example, if many words or subword pieces in the target language also appear in the source language, the model can represent the target text more effectively.
    - Tokenizer Quality Metrics: How well the model’s tokenizer handles the target language. Metrics like fertility (average number of subword tokens per word) and percentage of continued words (fraction of words split into multiple tokens) measure this quality. A high rate of broken-up words (high fertility or continued-word percentage) indicates the tokenizer struggles with the language, which negatively impacts performance on word-level tasks like POS tagging or NER. These metrics can be computed by tokenizing a sample corpus from the target language using the model’s tokenizer (e.g. with HuggingFace’s Tokenizers API).
    - Language Family or Geographic Features: Whether the target and source languages share a family or region. For instance, belonging to the same linguistic family (e.g. both Romance languages) or geographic proximity can sometimes serve as a proxy for hidden similarities (shared loanwords, etc.). In fact, research found that geographic/cultural proximity can be an informative feature for zero-shot transfer, on par with typological features.
    - Task Difficulty Features: Some features might describe the task itself in the context of the language, such as average sentence length or label set size, if relevant. For example, if the task is sentiment analysis, one might include the average number of words per sentence in the target language (as very long sentences could be harder). These are less common, but could be added if certain task properties are known to affect performance.
- Regression Models: Using these features, we train a regression model on any available data points (e.g. known performances from other languages or models). Simpler models like linear regression or Lasso (L1-regularized) regression can be effective and interpretable. Lasso can also perform feature selection by shrinking unimportant features to zero. Indeed, prior work used Lasso to find which factors most influence zero-shot performance. Tree-based ensemble models (e.g. Gradient Boosted Trees via XGBoost or Random Forests) are also popular for this kind of meta-prediction. For instance, XGBoost was used to assess feature importance in multilingual performance prediction tasks. These models can capture non-linear interactions between features (e.g. a model might learn that having both high token overlap and being in the same family yields an extra boost beyond each factor alone).
- Handling Different Scenarios with Features: In scenarios where the available reference point is very similar to the target (e.g. a dialect of the language or a model variant), the features will reflect that similarity (high lexical overlap, same family, etc.), leading the regressor to predict only a small drop in performance. Conversely, for distant languages/models, the features will indicate large differences (e.g. zero lexical overlap, different typology), and the regression should predict a lower performance. Essentially, a well-designed feature set and trained model allow extrapolating from known points to new combinations. For example, Lauscher et al. (2020) showed that even a simple regression on two features – pre-training corpus size and typological distance – can reasonably predict zero-shot performance for a multilingual model across languages. Subsequent studies extended this with richer feature sets (including morphological and lexical features) and more powerful regressors for better accuracy.

Tools: Implementing a feature-based predictor is feasible without heavy compute. You can use Python libraries like scikit-learn (for linear regression, Lasso, or RandomForestRegressor) or XGBoost (for gradient boosting) to train the model. The feature values can be gathered using resources such as lang2vec (for typological features), Wikipedia APIs or datasets (for rough text corpus size per language), and Hugging Face tokenizers (to compute subword tokenization metrics). All of these can be done on CPU. The data available for training such a regressor might come from public benchmarks like XTREME or XTREME-R, which provide performance of multilingual models on many languages for tasks like XNLI, POS tagging, NER, QA, etc. Even if each task has only ~10 known language points, combining them or using multi-task techniques (next section) can compensate for data sparsity.

## Multi-Task and Meta-Learning Approaches
A major challenge in performance prediction is the limited number of data points available per task. For instance, if we only have performance metrics for 5 or 10 languages for a given task, a naive regression might overfit or be unreliable. Multi-task learning addresses this by pooling information across multiple tasks or experimental settings to train a more robust predictor. The idea is to learn a shared representation that leverages commonalities in what influences performance, while still allowing task-specific differences. Here are key approaches and how they help:
- Joint Modeling of Tasks: Instead of training separate predictors for each task, we train a single model to predict performance for multiple tasks simultaneously. This can be done by having the model output a prediction for (task, language, model) triplets, using an input that includes task identity or task features in addition to language/model features. The model could be a multi-output neural network or an extension of the regression models mentioned above. For example, a multi-task Lasso regression (also known as a grouped Lasso) can be used to enforce that the model selects a common set of important features across all tasks. This was shown to identify a stable set of factors (like tokenizer quality, data size, etc.) that consistently influence zero-shot performance across various tasks.
- Advanced Multi-task Techniques: Researchers have explored techniques such as Collective Matrix Factorization and Multi-Task Gaussian Process (GP) Regression for this problem. In a matrix factorization view, one can imagine a matrix where rows are languages and columns are tasks (or model configurations), with entries being performance scores. Collective factorization can fill in missing entries by decomposing this matrix while also incorporating side features (like language or task embeddings). Gaussian Processes, on the other hand, can handle each task as a function to be learned with a kernel that shares information across tasks (the deep GP mentioned uses a neural network to learn an embedding for features before applying a multi-task GP). These methods are more complex but can yield better predictions in low-data regimes by borrowing strength from related data points.
- Meta-Learning (Few-Shot Learning for Tasks): Another angle is to use meta-learning algorithms like MAML (Model-Agnostic Meta-Learning) to train a predictor that can quickly adapt to a new task using only a few known points. In this context, each “task” for the meta-learner is a performance prediction problem for one NLP task. MAML would train a base model such that, given a few known language-performance pairs for a new task, it can fine-tune rapidly to predict other languages' performance. While powerful, meta-learning requires careful setup and might be overkill unless you expect to repeatedly face new tasks with minimal data.
- Benefits of Multi-task Approaches: Joint training tends to reduce overfitting and improve accuracy, especially for tasks where we have extremely limited direct data. A recent study reported about 10% reduction in prediction error overall by using multi-task learning compared to the best single-task model, with over 20% error reduction for tasks with very few known data points. In other words, a multi-task approach is particularly advantageous in the cold-start scenario where the target task has no or very few reference languages available. By learning general relationships (like “languages with tiny pretraining data tend to have low performance” or “tokenizer issues hurt POS tagging”), the model can make a more educated guess for a new language. For example, Ahuja et al. (2022) found that multi-task models predicted low-resource language performance more accurately, whereas separate single-task models sometimes did worse than even a simple average baseline in those cases. Multi-task training essentially acts as a form of regularization, guiding the predictor to rely on the most universally relevant features instead of spurious correlations from one task. Indeed, this approach revealed that certain features like subword overlap, tokenizer fertility, and pretraining size have consistent effects across many tasks.
- Incorporating Cross-Task Signals: If you do have some performance data on other tasks for the target language or model, those can be fed into a multi-task model as well. For example, imagine you know BERT’s accuracy on sentiment analysis and question answering for a language, and you want to predict its POS tagging accuracy. A multi-task model could treat sentiment, QA, POS as related tasks and possibly use the known scores as features or auxiliary outputs. Another simpler approach is to use the known task performance as an input feature for predicting the new task. (This is essentially a form of transfer learning between tasks in the prediction space.) Caution is needed: tasks that are very related (e.g. POS and NER are both syntactic/labeling tasks) are more likely to have correlated performance than very dissimilar tasks. Research by Dolicki & Spanakis (2021) found that no single feature or factor predicts performance well for all tasks, underscoring that each task might emphasize different asp. Therefore, multi-task models may internally learn to weight features differently per task. For practitioners, the takeaway is that if you have access to cross-task performance data, use it – either via joint modeling or by constructing a richer feature set – but also account for task differences (perhaps by clustering similar tasks or including a “task type” feature).

Tools: Many multi-task strategies can still be implemented on a modest machine. For example, scikit-learn’s MultiOutputRegressor or MultiTaskLasso can handle simple multi-task regression. If you prefer neural networks, frameworks like PyTorch (with careful coding) or high-level libraries (Keras/TensorFlow) can train a small multi-layer perceptron that takes features plus maybe a task ID (one-hot) and outputs a predicted score. Given the small dataset sizes (a few tasks and tens of languages), these models train very quickly on CPU. For Gaussian Process regression, libraries like GPy or sklearn.gaussian_process could be tried, though a multi-task GP might require a bit more customization. If using matrix factorization, you could use libraries like pandas or numpy to implement a simple solver or leverage recommender system toolkits (though often just solving via scikit-learn’s Ridge on an augmented feature matrix might suffice). The key is that these advanced methods are optional – you can start with simpler regression and only consider multi-task learning if single-task predictions are too unstable.

## Extrapolating to New Model Sizes and Architectures
Sometimes the change is not in language, but in the model. Perhaps we want to predict performance for a larger model given results from a smaller one, or for a new model family given an older model’s performance. Different model sizes or architectures can have systematic performance differences that we can exploit in predictions:
- Model Scaling Laws: It has been empirically observed that increasing model size (parameters) or training data often yields predictable improvements following a power-law or log-linear trend. Kaplan et al. (2020) famously demonstrated smooth scaling laws for language models, where performance (e.g. perplexity or accuracy) improves as a power-law function of model compute, size, and data volume. In practical terms, if you have performance metrics for a smaller model, you can extrapolate to a larger model by assuming such a trend. For instance, if a BERT-base (110M parameters) achieves X% on a task, a BERT-large (340M) might achieve a few points higher. The exact delta can be estimated by fitting a function on any known points (if you have two or three sizes, fit a power-law curve; if only one, you might rely on published scaling trends). Even without heavy math, one rule of thumb is that error often decreases roughly as a power of model size (diminishing returns). So doubling parameters might yield, say, a ~√2 factor improvement in the remaining error (this is just an illustration; actual exponents vary per task). The key is that bigger models rarely worsen performance in NLP tasks (unless overfitting on tiny data), so you can confidently predict "same or better performance" and quantify the gain from known benchmarks. Using this approach in the same language, different size scenario can be as simple as: if the model is 4 times bigger and all else is similar, expect an improvement corresponding to known scaling behavior. Academic papers or community leaderboards often provide the performance of multiple model sizes on the same task, which you can use as a reference to calibrate your prediction.
- Cross-Model Calibration: For different model families (e.g., predicting how an XLM-Roberta model will perform given BERT performance, or how GPT-style will do given BERT-style), we can incorporate model-specific features and historical deltas. One straightforward method is to include features describing the model architecture itself in your regression. Examples: the model type (Transformer vs LSTM, or “encoder-only vs decoder vs seq-to-seq”), number of layers, parameter count, vocabulary size, training objective (masked language modeling vs autoregressive), etc. By training your performance predictor on data from multiple models, it can learn, for example, that “Model = XLM-R” tends to outperform “Model = mBERT” by a certain margin on low-resource languages, possibly due to more training data or a better tokenizer. In fact, research comparing multilingual BERT to XLM-R found that XLM-R (trained on more data with a SentencePiece tokenizer) usually yields higher scores, especially on languages with limited resources. If you know such qualitative differences, you can bake them into the prediction. For instance, if mBERT gets 60 F1 on a low-resource language NER, XLM-R might get around 65 F1 under the same conditions, owing to its larger training corpus – your model feature could capture this expected increase. Another example: monolingual models often outperform multilingual ones on their specific language (due to no capacity dilution), so if you’re predicting how a monolingual model would do given a multilingual model’s performance, you might add a few points. These adjustments can be done via a learned model or simple rules based on known evaluations.
- Example – Using Known Differences: Suppose we have the same language and task evaluated with two model families: a sequence-to-sequence model and a transformer encoder. If the seq2seq historically lags behind on this task by, say, 5 points, then if the transformer got 80%, we predict seq2seq ~75%. Without such direct data, rely on model features: e.g., include a binary feature “Has cross-lingual pretraining” – models that were pretrained on multiple languages (like mBERT, XLM-R) may have an edge in zero-shot transfer compared to ones pretrained only on English. Another example of a feature is the tokenizer type or vocabulary coverage: different models have different tokenization. If the target language uses a script not covered well by Model A’s tokenizer but Model B’s tokenizer handles it (maybe Model B has more unicode support or a larger vocab), this could predict that Model B will excel where Model A struggles. This ties into the tokenizer metrics mentioned earlier, which can be computed per model for the target language.
- When No Direct Comparison Data: In a scenario where same model & same language are both new (cold start), you might combine the above strategies. For instance, use known data from other languages for that model and from that language with other models. If you know how the model generally performs relative to another model (from a different language), and you know how the other model performs on the target language (from a different context), you can compose these pieces. This is essentially a form of transferable calibration. It may not be perfectly accurate, but it gives a starting point. For example: Model X is usually 5 points better than Model Y on language Z (observed on a similar task). We have Model Y’s performance on our target language; thus we predict Model X’s performance on target is ~5 points higher. Such additive or multiplicative adjustments could be refined by your regression model if you include interaction features like (Model_X vs Model_Y) along with language features.


Tools: Implementing scaling law extrapolation can be as simple as using NumPy/Pandas to fit a curve. If you have multiple model sizes, you can use numpy.polyfit on log-scale data or do a simple regression in log-space. For including model descriptors in your feature-based model, just extend your feature vector (e.g., add a one-hot encoding for model type or a numeric feature for parameter count). Ensuring these features are normalized (e.g., log10 of parameter count, since raw counts vary by orders of magnitude) is helpful. No GPU is needed – these are lightweight computations. If you have access to training curves or validation scores of the model on similar tasks, you can even use that (though in pure zero-shot scenario, you might not).
Lastly, it’s worth noting that large changes in architecture (like going from a bag-of-words classifier to a Transformer) might not have a simple predictable relation – in such cases, try to gather at least some reference points for the new architecture on any language to see its general level, and use that alongside your predictions. But within modern transformer families, these methods work quite well because most state-of-the-art models improve on their predecessors in known ways.
## Exploiting Cross-Task Information
Thus far, we assumed the task is fixed and we vary language or model. But what if the task itself is new, and the only data you have is from other tasks? This is a common scenario – for example, you might know a model’s performance on translation and parsing, and need to predict its performance on, say, summarization in the same language. While this is a harder leap (because tasks can differ greatly), there are ways to leverage cross-task information:

- Task Feature Vectors: Just as languages can be described by features, tasks can be too. You can create features that characterize the task’s nature – e.g., is it classification vs sequence labeling vs generation? Does it require world knowledge (open-domain QA) or just syntax (part-of-speech tagging)? If you have a meta-dataset of tasks, you could include task embeddings (perhaps derived from a taxonomy or from descriptions). This is advanced and often not available, so a simpler proxy is to categorize tasks (semantic task vs syntactic task, etc.) and include a categorical feature. In performance prediction research, it has been observed that syntactic tasks (like POS tagging, NER) often rely more on language similarity features, whereas semantic tasks (like NLI or QA) might be influenced more by language understanding capacity and data size. Including a task type indicator helps the model adjust which features to pay attention to for the given task.
- Using One Task’s Performance to Predict Another’s: If you have the model’s results on a different task in the target language, you can use that as a strong hint. For instance, if a model performs poorly on all other tasks for that language, it’s likely it will also struggle on the new task (and vice versa). A simple approach is a linear regression mapping performance from one task to another. If historically across languages, Task B’s accuracy is usually, say, 0.8 × Task A’s accuracy (maybe NER is 80% of POS, hypothetically), you can use Task A’s known score for the language to predict Task B. Alternatively, include Task A’s score as an input feature in a multi-task regression model. The multi-task learning approach described earlier inherently does this: it learns correlations between tasks’ performance. In the absence of a multi-task model, you can manually create features like “Model’s average performance on other tasks in this language” or even specific ones like “NER F1 in this language” if available, to plug into your predictor for, say, QA F1. This essentially treats the model’s other task success as an indicator of general language proficiency.
- Knowledge Transfer via Scaling or Adjustment: If the task has known difficulty differences, you can adjust from a reference task. For example, if we know that in general a model gets 5 points lower on task T2 than on task T1 in the same language (perhaps translation is harder than classification by a fixed margin), then take the known T1 score and subtract 5. This requires strong assumptions and is risky if not backed by data, but in scenarios with absolutely no direct info on T2, it's an option to consider. Usually, it's better to rely on actual data through a learned model rather than hard-coding differences, because the gap between tasks can itself vary by language or model.
- Scenario Example: Let's say we have no direct data for document summarization in Swahili with our model, but we do know how the model does on sentiment analysis in Swahili. If sentiment accuracy is low (indicating the model struggles with Swahili in general), we might predict summarization will also be relatively low, even if we haven’t tested it. On the other hand, if the model excels at sentiment in Swahili, it means the language isn’t a barrier for that model, so summarization might also be decent (though possibly a bit lower if summarization is inherently harder). By training on other languages where we have both tasks, a multi-task predictor could formalize this intuition. In practice, Ahuja et al. (2022) demonstrated exactly this benefit: by jointly training on 11 tasks, they could predict a task with missing language data more reliably using the patterns learned from other tasks. They also found that certain features like tokenizer issues mattered mainly for specific tasks (word-level ones) and not others, which the multi-task model could discern.


Tools: If you plan to use other task performance as features, you’ll need that data collected. This might involve running the model on some easier-to-obtain evaluation sets in the target language (if available) as a preliminary step. For instance, unsupervised tasks or proxy evaluations could be done: maybe evaluate the model’s perplexity on a sample of target language text – this is not a “task performance” per se, but it’s a measure of language modeling ability which correlates with many task performances. Tools like Hugging Face Transformers allow you to compute perplexity or zero-shot classification scores easily and on CPU for moderate model sizes (or use smaller distilled models to gauge). For multi-task model building, the same libraries mentioned earlier apply. If you have to encode task properties, you might manually create a dictionary of task types or use embeddings from a library (there’s no off-the-shelf library for task embeddings as commonly as for language features, so this part is usually manual or learned from data). A practical approach: treat task as just another categorical feature and use one-hot encoding or learn an embedding for it inside a neural network model.
## Large Language Models in Zero-Shot Settings
No guide on modern NLP would be complete without discussing Large Language Models (LLMs) in zero-shot or few-shot use. LLMs (like GPT-3, GPT-4, PaLM, etc.) are trained on massive corpora and can perform many tasks via prompting, often across languages, without fine-tuning. How do they fit into performance prediction? There are a few angles to consider:

- LLMs as the Model to Evaluate: If the task will be done by prompting an LLM (instead of fine-tuning a model), you might want to predict its performance on the task for the target language. LLMs tend to have strong performance on languages present in their training data, but can be weaker on low-resource languages or those with different scripts. For example, an English-centric LLM might perform nearly as well in Spanish or French (which share a script and were in training data) but could falter in Swahili or Burmese if those languages were scarcely seen. If there’s no published evaluation of the LLM on your specific task and language, you can use a combination of language features (does the LLM know the language?) and task type considerations. Many LLMs are surprisingly good at tasks that involve general knowledge or reasoning in any language they understand, but could struggle with tasks requiring precise linguistic understanding if the language is unfamiliar. One practical step is to use the LLM itself in a self-evaluation way: you can prompt the LLM on a handful of known test questions in the target language (if you can translate a few examples) and observe how it does. This can be done without a GPU by calling an API or using a small open-source LLM on CPU (for example, BLOOM 560M or GPT-Neo small models can run on CPU to get a rough sense, though they are weaker than GPT-3). This kind of quick empirical check can inform your prediction.
- Combining LLMs with Structured Methods: LLMs can also assist in feature generation. For instance, you could prompt an LLM to guess the difficulty of the task in the target language or to identify similarities between languages in descriptive terms. However, since our goal is a systematic guide, it’s safer to rely on measurable features. One could imagine using an LLM to generate synthetic data or intermediate metrics (like translate some text to a pivot language and measure quality as a proxy for how well the model understands the target language). These are advanced strategies and may not be necessary if you have the features and methods described above.
- When to Use LLM vs Fine-tuned Model: Sometimes the “prediction” might inform a decision: should I use a fine-tuned smaller model or just query an LLM for this task and language? If your features suggest that a fine-tuned multilingual model would only achieve, say, 50% accuracy due to the language being very low-resource, but you know an LLM has probably seen some of that language, the LLM might outperform in zero-shot (LLMs with tens or hundreds of billions of parameters have broad multilingual knowledge). In such cases, the performance prediction guide can help decide the approach. There have been studies where a zero-shot GPT-3 was competitive with task-specific models on certain benchmarks, demonstrating that with no training, an LLM can sometimes reach decent performance. But you should also weigh that LLM prompting might be more costly (and the performance can be unpredictable if prompt is not well-crafted).
- Features for LLM’s multilingual ability: For an LLM, you might not have an explicit “pretraining data size per language” available (unless the model paper provided it). But proxies exist: for example, if it’s known that the target language is present in CommonCrawl or Wikipedia, it’s likely in the LLM’s training. Or one can use perplexity of the LLM on a sample of target text as a feature – if the perplexity is low, the LLM is comfortable in that language, suggesting good performance; if it’s sky-high, the LLM is struggling to even parse it. Tools like the Hugging Face evaluate library or custom scripts can measure perplexity by feeding the model with the target language text.

In summary, LLMs bring a new dimension to zero-shot performance: they often have the capability built-in, so the prediction might be directly confirmed by a quick test. If using an API like OpenAI’s, you could simply run a few examples and measure accuracy, which for a small sample might suffice as a performance estimate (since we assume no heavy compute for big evaluations). This is a departure from earlier sections where the model is fixed and smaller – but it’s an available method if one has no other information. Nonetheless, for consistency and reproducibility, relying on the structured approaches (feature regression, multi-task models) provides a clear rationale for the prediction.
## Practical Tips and Tools Summary
To wrap up, here is a concise list of practical tips, features, and Python tools you can use when implementing performance prediction for a new language or model task:
- Gather Language Features: Use lang2vec or the URIEL database for typological features (e.g. does the language use articles? What is its word order? etc.). Use Wikipedia or Common Crawl stats for corpus size. You can get Wikipedia article counts via their API or from published lists. For lexical overlap, collect a vocabulary sample of the target language (from an unlabeled corpus or even a translation dictionary) and compare it to the fine-tuning language vocab. A quick Python approach is to use fastText pretrained word vectors: if you have vectors for both languages, the cosine similarity of frequent word vectors or the overlap of nearest neighbors can serve as a lexicon similarity feature.
- Compute Tokenizer Metrics: Load the model’s tokenizer with Hugging Face Transformers and tokenize a representative text in the target language. Calculate the average number of tokens per word and the fraction of words that got split into multiple tokens (PCW). These numbers directly give fertility and continued-word metrics. They require only the tokenizer and some text (which you can obtain from any raw corpus in that language, even a Wikipedia dump or news article). No GPU needed, just CPU for tokenization.
- Build the Feature Table: For each known data point (e.g. a tuple of (Language, Model, Task) with a performance result), create a row of features as discussed. Also prepare a row for your target (Language, Model, Task) with those features. Ensure to normalize features (e.g. scale them 0-1 or take logs as appropriate) similarly to how your training data is scaled.
- Choose a Prediction Algorithm: Start simple with LinearRegression or Ridge from scikit-learn to see if a linear combination of features suffices. If nonlinear effects seem important, try RandomForestRegressor or XGBRegressor (XGBoost) – these handle nonlinearity and interactions automatically. Set aside some known points as a validation to gauge accuracy of your predictor. If doing multi-task, you can either use scikit-learn’s MultiTaskLasso (which solves a linear model for all tasks jointly) or train a small neural network with multiple outputs (one per task). Libraries like scikit-learn even allow you to do multi-output regression with trees and forests (e.g. RandomForestRegressor(n_outputs=...) can predict multiple outputs). This could be useful if you treat each task as an output dimension.
- Evaluate and Interpret: Check which features the model relied on. For linear models, coefficients will tell you (e.g. it might put a large positive weight on “log(pretrain_data_size)” and a negative weight on “tokenizer_fertility”). For tree-based models, use feature_importances_ or better yet SHAP (SHapley Additive exPlanations) values to see feature impact. Prior work found, for example, that subword overlap and tokenizer continuation had high importance for some tasks, while genetic (family) similarity might matter less once other features are accounted for. Understanding this can guide you to refine features or trust the model’s basis for prediction.
- Addressing Uncertainty: Remember that these predictions have uncertainty. It can be useful to predict not just a point estimate but a range. Techniques like Gaussian Process regression inherently give a variance for the prediction, but you can also empirically judge error from your validation. If your predictor was often off by, say, ±2 points in F1, you can give a confidence interval in your final estimate. This is practical when communicating to stakeholders – e.g. “We estimate ~70% accuracy, give or take a few points.” If using an ensemble of models (another way to improve reliability), you could take the average prediction and see the spread.
- When All Else Fails – Do a Trial: If the cost of actually evaluating the model on a small sample in the target language isn’t prohibitive, nothing beats a direct test. Our methods are to avoid heavy training, but running the model inference on some labeled data (if you can obtain or translate 100 test examples by hand) can provide a sanity check for your predictions. You can use that tiny evaluation to adjust your model if needed (in effect, one-shot calibration). Since we assumed no GPU, this would mean using the model in inference mode on CPU, which for moderate model sizes and a small number of examples is usually fine (just slower). For example, you could take a multilingual BERT model fine-tuned on English, and run it on 50 translated sentences in the target language to see how it does – if it gets half right, that’s 50% accuracy, which either validates or updates your expectations.
